# GitHub Actions Setup Guide

This document explains how to set up the arceion-zorion package for automated publishing via GitHub Actions.

## Quick Start

### 1. Create PyPI Account & API Token

1. Sign up at [PyPI](https://pypi.org/account/register/) (if you don't have an account)
2. Go to [Account Settings → API tokens](https://pypi.org/manage/account/tokens/)
3. Click "Add API token"
4. Name it `arceion-zorion-github`
5. **Copy the token immediately** (displayed only once)
6. Format: `pypi-AgEIcHlwaS5vcmc...`

### 2. Add GitHub Secret

1. Go to your repository on GitHub
2. Click **Settings** → **Secrets and variables** → **Actions**
3. Click **New repository secret**
4. Name: `PYPI_API_TOKEN`
5. Paste the token you copied
6. Click **Add secret**

### 3. Verify Workflows

All workflows are in `.github/workflows/`:

- **tests.yml** — Runs pytest on Python 3.10, 3.11, 3.12, 3.13
- **build.yml** — Builds distributions (wheel + source)
- **publish.yml** — Publishes to PyPI (triggered by releases)

They will automatically run on:
- ✅ Push to `main`, `master`, `develop`
- ✅ Pull requests
- ✅ Release creation (auto-publish)
- ✅ Manual workflow dispatch

### 4. Test the Setup (Optional)

To verify everything works **before** publishing:

1. Go to your repository → **Actions** → **Publish to PyPI**
2. Click **Run workflow**
3. Select `testpypi` from dropdown
4. Click **Run workflow**

This will:
- Run tests
- Build package
- Publish to [TestPyPI](https://test.pypi.org/simple/)

Then test installation:
```bash
pip install --index-url https://test.pypi.org/simple/ arceion-zorion==0.1.0
```

### 5. Release to Production

When ready to release:

1. Update version in `pyproject.toml`
2. Commit: `git add pyproject.toml && git commit -m "chore: bump version"`
3. Push: `git push origin main`
4. Go to **Releases** → **Draft a new release**
5. Tag: `v0.2.0` (matches pyproject.toml version)
6. Fill in release notes
7. Click **Publish release**

The workflow automatically publishes to PyPI! ✅

## Workflows Explained

### tests.yml

**Triggers:** Push, PR, manual dispatch
**Jobs:** Test on Python 3.10, 3.11, 3.12, 3.13

Runs:
```bash
pytest arceion/zorion/tests/ -v --cov=arceion/zorion
```

**Status:** Optional failures allowed (continues on error)

### build.yml

**Triggers:** Push, PR
**Jobs:** Build on Python 3.10, 3.11, 3.12

Runs:
```bash
python -m build          # Creates wheel + source distribution
twine check dist/*       # Validates metadata
```

### publish.yml

**Triggers:** Release creation (auto) or manual workflow_dispatch

**Steps:**
1. ✅ Test (must pass)
2. ✅ Build (must pass)
3. ✅ Publish (if secrets configured)

**Publish targets:**
- `pypi` — Production (default for releases)
- `testpypi` — Testing (manual workflow only)

## Troubleshooting

### "Workflow file not recognized"

**Fix:** Ensure `.github/workflows/publish.yml` exists and is committed:
```bash
git status
git add .github/workflows/
git commit -m "Add GitHub Actions workflows"
git push
```

### "Error: Unable to find 'python-build'"

**Fix:** The workflow handles this. Just ensure `pyproject.toml` exists.

### "Workflow shows as draft but won't run"

**Fix:** Workflows only run after commits are pushed. If it's in a PR:
- PR must be from same repo (not fork)
- Branch must be listed in workflow triggers

### "ModuleNotFoundError: No module named 'build'"

**Fix:** Workflow installs build automatically:
```bash
python -m pip install build twine setuptools wheel
```

If local test fails, run:
```bash
pip install build twine
```

### "Authentication failed — Invalid token"

**Fix:**
1. Token might be expired or wrong format
2. Regenerate on PyPI: [Account Settings → Tokens](https://pypi.org/manage/account/tokens/)
3. Update GitHub secret with new token
4. Retry workflow

### "Package name not available"

**Fix:**
- Package name must be unique on PyPI
- Check: `pip search arceion-zorion` or [pypi.org/project/arceion-zorion/](https://pypi.org/project/arceion-zorion/)
- If taken, change name in `pyproject.toml`

## Security

✅ **Secrets are safe:**
- GitHub Actions encrypts secrets
- Secrets never appear in logs
- Token is only used in publish step
- Only repository maintainers can access secrets

✅ **No credentials in code:**
- `.env` files in `.gitignore`
- No tokens in source files
- API keys loaded from GitHub Secrets

✅ **All code is reviewed:**
- Workflows are version controlled
- Changes require commits/PRs
- Runs are auditable in Actions tab
